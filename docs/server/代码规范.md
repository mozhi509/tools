# Server 代码规范

后端（Express + TypeScript）代码风格与结构约定。

---

## 一、技术栈

- **运行时**: Node.js
- **框架**: Express 4
- **语言**: TypeScript（编译到 `dist/`）
- **存储**: Redis（会话、分享、聊天等）
- **安全**: helmet、cors、express-rate-limit

---

## 二、目录结构

```
server/
├── routes/           # 按业务拆分的路由模块
│   ├── tools.ts      # 工具类 API（JSON、Base64、URL 等）
│   ├── video.ts      # 视频上传与处理
│   ├── share.ts      # 分享短链
│   ├── chat.ts       # 聊天会话与消息
│   └── *.d.ts        # 路由相关类型（若需要）
├── types/            # 公共类型定义
│   ├── express.d.ts  # 请求/响应体类型
│   └── video.ts      # 视频相关类型
├── redis.ts          # Redis 连接与封装
└── index.ts          # 入口、中间件挂载、启动
```

- 新增业务：在 `routes/` 下新建 `xxx.ts`，在 `index.ts` 中 `app.use('/api/xxx', xxxRouter)`。
- 请求/响应体类型放在 `types/`，或在路由文件顶部用 `interface` 定义。

---

## 三、入口与中间件（index.ts）

- 中间件顺序：安全(helmet) → 限流(rateLimit) → cors → body 解析 → 静态资源(生产) → 路由 → 错误处理。
- 生产环境才挂载静态目录、`get('*')` 回退到前端；开发环境由前端 dev server 代理 `/api`。
- 设置 `app.set('trust proxy', true)`，以便在反向代理后正确取 `req.ip`。
- 错误处理中间件签名为 `(err: unknown, req, res, next)`，内部用 `err instanceof Error` 再访问 `message`/`stack`，避免把非 Error 当 Error 用。

---

## 四、路由与 handler

### 4.1 路由组织

- 每个模块：`const router = Router();`，导出 `export default router`。
- 使用 `router.get/post/put/delete(...)` 注册；路径与前端 `API_ENDPOINTS` 一致（如 `/api/tools` 挂载后，路由写 `/json/format` 即完整路径 `/api/tools/json/format`）。

### 4.2 请求与响应类型

- 使用 Express 泛型标注请求体与响应体，例如：

```ts
import { Request, Response } from 'express';

router.post('/json/format', (
  req: Request<{}, JsonFormatResponse, JsonRequest>,
  res: Response<JsonFormatResponse>
) => { ... });
```

- 路径参数：`Request<{ chatId: string }>`，通过 `req.params.chatId` 使用。
- 请求体：从 `req.body` 读取，先校验必填字段再处理；响应统一 `res.json({ ... })`，成功可包一层 `{ success: true, data: ... }`。

### 4.3 错误处理

- **禁止**在 handler 里使用 `catch (error: any)`；统一使用 `catch (error: unknown)`。
- 需要向客户端或日志暴露错误信息时，先做类型收窄：

```ts
catch (error: unknown) {
  const message = error instanceof Error ? error.message : String(error);
  res.status(400).json({ success: false, error: message });
}
```

- 生产环境日志可只打 `message`，开发环境可打完整 `error` 或 `stack`，避免把敏感信息返回给前端。

## 五、Redis

- 连接与封装在 `redis.ts`：`connectRedis()`、`disconnectRedis()`、导出 `redisClient`。
- 各路由按需在 handler 内 `await connectRedis()`，再使用 `redisClient.get/set/setEx/lRange/rPush/lTrim/exists` 等。
- Key 命名建议带业务前缀，如 `share:${id}`、`chat:session:${chatId}`、`chat:messages:${chatId}`；有过期需求的使用 `setEx` 或 `expire`。
- 配置从环境变量读取：`REDIS_HOST`、`REDIS_PORT`、`REDIS_PASSWORD`、`REDIS_DB`；`redis.ts` 内用显式类型描述 config 对象，避免 `any`。

---

## 六、类型定义

- 请求体、响应体接口放在 `types/express.d.ts` 或各路由文件顶部；跨文件复用的放在 `types/xxx.ts`。
- 避免在业务代码中使用 `any`；暂时无法精确类型时可用 `unknown` 再断言或收窄。
- 与第三方库（如 ffmpeg）交互时，优先使用库自带的类型（如 `ffmpeg.FfprobeData`），避免手写不一致的接口导致类型错误。

---

## 七、安全与校验

- 用户输入：校验必填、类型、长度；文件上传校验 MIME、大小，并做路径规范化，防止路径遍历。
- 生产环境错误响应不返回堆栈或内部细节，仅返回通用提示（如「服务器内部错误」）。
- 敏感配置（如 Redis 密码、域名）从环境变量读取，不写死在代码中。

---

## 八、命名与注释

- **文件名**：小写 + 连字符仅用于多词（如无则小写），如 `index.ts`、`redis.ts`、`share.ts`。
- **变量/函数**：camelCase；常量可用 UPPER_SNAKE_CASE 或普通 const。
- **接口/类型**：PascalCase，如 `JsonRequest`、`ChatMessage`。
- 路由文件顶部可对模块职责做简短注释；复杂逻辑或安全相关处理加注释说明意图。

---

## 九、其他

- 构建：`tsc` 以项目根目录 `tsconfig.json` 为准，`rootDir` 为 `./server`，输出到 `dist/`。
- 不提交调试用 `console.log`；生产日志保持简洁、可追踪。
- 新增环境变量时，在 `.env.example` 中补充说明，便于部署与协作。
